## Executive Summary

The 06.1.2 pipeline delivered a **functional** smart playlist authoring system: models, ViewModel, creation/edit forms, live preview, rule editing, template picker, analytics view, and UserDefaults persistence. However, a gap analysis reveals several UX and robustness issues that prevent this from being a **complete, production-quality** authoring experience. This plan targets those gaps while validating existing behavior meets all acceptance criteria.

## Implementation Steps

### Phase 1: Detail View Refresh Fix (G4) — Critical Path

**Rationale**: The acceptance criterion "reflect updates in detail view" is blocked if the detail view shows stale data after editing.

1. **Verify the refresh behavior** by reading the `navigationDestination(for: SmartPlaylistNavigation.self)` closure in `PlaylistViews.swift`. The closure reads `smartVM.smartPlaylists.first(where:)` which is `@Observable` state. When `updateSmartPlaylist` calls `reload()`, the `smartPlaylists` array changes, which should trigger SwiftUI to re-evaluate the destination closure with the updated value.

2. **If the re-evaluation works correctly** (the `let smartPlaylist` parameter gets the new value because the closure re-runs), no code change is needed — just add a test to confirm.

3. **If the detail view retains the stale value** (NavigationStack caches the destination), refactor `SmartPlaylistDetailView` to accept a `smartPlaylistID: String` and look up the playlist from the ViewModel directly:
   ```swift
   // Before (stale reference risk):
   let smartPlaylist: SmartEpisodeListV2

   // After (always reads latest):
   let smartPlaylistID: String
   private var smartPlaylist: SmartEpisodeListV2? {
       viewModel.smartPlaylists.first { $0.id == smartPlaylistID }
   }
   ```

### Phase 2: Rule Authoring Enhancements (G1, G3)

4. **Add rule reordering** to `SmartPlaylistCreationView`:
   - Add `.onMove` modifier to the `ForEach(rules)` block
   - Add `EditButton()` in the toolbar (secondary placement, iOS only)
   - Implementation:
     ```swift
     .onMove { source, destination in
         rules.move(fromOffsets: source, toOffset: destination)
     }
     ```
   - The `.onMove` will trigger `.onChange(of: rules)` → `refreshPreview()` automatically.

5. **Add rule duplication** via context menu on each `SmartPlaylistRuleRow`:
   - Wrap `SmartPlaylistRuleRow` in a view that provides a `.contextMenu` with "Duplicate Rule"
   - On tap, create a new `SmartListRule` with same type/comparison/value/negation but new UUID, insert after the current rule
   - Implementation:
     ```swift
     .contextMenu {
         Button {
             let copy = SmartListRule(
                 type: rule.type,
                 comparison: rule.comparison,
                 value: rule.value,
                 isNegated: rule.isNegated
             )
             if let idx = rules.firstIndex(where: { $0.id == rule.id }) {
                 rules.insert(copy, at: idx + 1)
             }
         } label: {
             Label("Duplicate Rule", systemImage: "plus.square.on.square")
         }
     }
     ```

### Phase 3: Unsaved Changes Confirmation (G2)

6. **Track whether the form has unsaved changes** in `SmartPlaylistCreationView`:
   - Add computed property comparing current state to initial state:
     ```swift
     private var hasUnsavedChanges: Bool {
         if let existing = existingSmartPlaylist {
             return name != existing.name
                 || description != (existing.description ?? "")
                 || rules != existing.rules.rules
                 || logic != existing.rules.logic
                 || sortBy != existing.sortBy
                 || maxEpisodes != existing.maxEpisodes
                 || autoUpdate != existing.autoUpdate
                 || refreshInterval != existing.refreshInterval
         } else {
             // For new playlists, check if user has entered anything beyond defaults
             return !name.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty
                 || !description.isEmpty
                 || rules.count > 1
                 || rules.first?.type != .playStatus
         }
     }
     ```
   - Add `@State private var showingDiscardConfirmation = false`
   - Replace Cancel button action:
     ```swift
     Button("Cancel") {
         if hasUnsavedChanges {
             showingDiscardConfirmation = true
         } else {
             dismiss()
         }
     }
     ```
   - Add `.confirmationDialog`:
     ```swift
     .confirmationDialog("Discard Changes?", isPresented: $showingDiscardConfirmation) {
         Button("Discard", role: .destructive) { dismiss() }
         Button("Keep Editing", role: .cancel) { }
     } message: {
         Text("You have unsaved changes. Are you sure you want to discard them?")
     }
     ```

## Architectural Decisions

### ADR-1: Keep Inline Rule Editor (No Advanced Sheet)
**Decision**: Enhance the existing `SmartPlaylistRuleRow` inline editor rather than integrating the `SmartListRuleBuilderView` from LibraryFeature.
**Rationale**: The LibraryFeature builder uses `ObservableObject`/`@StateObject` while PlaylistFeature uses `@Observable`. Mixing observation systems adds complexity. The inline editor already supports all 13 rule types with appropriate value editors. A sheet-per-rule flow would add navigation friction for simple edits.
**Tradeoff**: Advanced features like duration sliders with quick presets are deferred. Can be added to the inline editor later if needed.

### ADR-2: Defer SmartPlaylistManagerAdapter Wiring
**Decision**: Continue using `UserDefaultsSmartPlaylistManager` at the app layer rather than wiring `SmartPlaylistManagerAdapter`.
**Rationale**: `UserDefaultsSmartPlaylistManager` already provides working persistence. The adapter adds async bridging and timer-based auto-refresh, but `SmartPlaylistViewModel` doesn't need these features — it triggers evaluation synchronously via `allEpisodesProvider`. Wiring the adapter is a separate concern for 06.1.2.2+ when auto-refresh becomes more important.
**Risk**: Low. Both managers conform to `SmartPlaylistManaging` and are interchangeable.

### ADR-3: Commutative Rule Evaluation
**Decision**: Rule order does NOT affect evaluation results (AND/OR are commutative operators). Reordering is a UX convenience for authoring organization, not a semantic change.
**Rationale**: The `DefaultEpisodeFilterService.evaluateSmartListV2` processes all rules in the set without short-circuit ordering. This matches user expectations from music app smart playlists.

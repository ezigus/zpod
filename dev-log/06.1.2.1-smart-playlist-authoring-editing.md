## Changes Made

### SmartPlaylistViews.swift

**G4 ‚Äî Detail view refresh fix (Critical)**
- Added `currentPlaylist: SmartEpisodeListV2` computed property that reads from `viewModel.smartPlaylists` (the `@Observable` state) rather than the captured `let smartPlaylist` value.
- All body references updated to `currentPlaylist` so edits are reflected immediately after sheet dismissal.
- Key insight: since `viewModel` is `var` (observed), SwiftUI tracks `viewModel.smartPlaylists` access during body evaluation and triggers re-render when `reload()` updates the ViewModel post-edit.

**G1 ‚Äî Rule reordering**
- Added `.onMove` modifier to `rulesForEach` (iOS only, using `#if os(iOS)` branch in `@ViewBuilder` property).
- Section header shows "Reorder / Done" toggle button (iOS only) that activates `.environment(\.editMode, .constant(.active))`.
- `isReordering` state hides "Add Rule" button during reordering to prevent layout confusion.
- Platform note: `\.editMode` and `.onMove` are both `@available(macOS, unavailable)`. Used `@ViewBuilder rulesForEach` property with `#if os(iOS)` to cleanly separate platforms without duplicating row content.

**G2 ‚Äî Unsaved changes confirmation**
- `hasUnsavedChanges` computed property compares all form fields (name, description, rules, logic, sortBy, maxEpisodes, autoUpdate, refreshInterval) against the saved playlist state.
- Cancel button now shows `.confirmationDialog("Discard Changes?")` with "Discard" (destructive) and "Keep Editing" (cancel) actions when `hasUnsavedChanges == true`.

**G3 ‚Äî Rule duplication**
- `.contextMenu` on each `SmartPlaylistRuleRow` (via `ruleRow(for:)` helper) provides "Duplicate Rule" action.
- Creates a new `SmartListRule` with same type/comparison/value/negation but new UUID, inserted immediately after the source rule.

**G7 ‚Äî Preview section match count**
- Preview section now has a custom header with "Preview" label + "N matches" badge (blue, caption weight) when episodes are present.
- Removed the in-body count text row that was less visually prominent.

**G8 ‚Äî String rule validation hint**
- `SmartPlaylistRuleValueEditor` now wraps `.podcast`, `.title`, `.description` cases in a `VStack` that shows an orange advisory `Text` when the string value is empty.

## Architectural Decisions

### ADR-1: Keep Inline Rule Editor
No change. The existing `SmartPlaylistRuleRow` inline editor covers all 13 rule types. The `SmartListRuleBuilderView` in LibraryFeature uses `ObservableObject`/`@StateObject` while PlaylistFeature uses `@Observable` ‚Äî mixing would add complexity without user-facing benefit.

### ADR-2: Defer SmartPlaylistManagerAdapter Wiring (G9)
`UserDefaultsSmartPlaylistManager` already provides working persistence verified by `testCustomPlaylistSurvivesReloadFromStorage` and related Persistence package tests. The adapter adds timer-based auto-refresh; this is relevant for 06.1.2.2+ when background sync becomes a requirement.

### ADR-3: Rule Order is Commutative
`testRuleOrderIsCommutativeForEvaluation` confirms AND/OR evaluation is order-independent. Reordering is purely organizational.

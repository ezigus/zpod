# Dev Log: 27.1.9 — Wire Library View to Live Podcast Repository Data

**Branch**: `feat/-27-1-9-wire-library-view-to-live-podcas-426`
**GitHub Issue**: #426
**Status**: Complete

---

## Intent

Surface the persistent `PodcastManaging` data in the Library tab. The library was rendering hardcoded `samplePodcasts` instead of calling the live repository that was wired in issue 27.1.2.

## Root Cause

`LibraryView` never received the `podcastManager` dependency. `ContentView` held a `podcastManager` instance but only passed `playlistManager` down to `LibraryView`. The view used a local `PodcastItem` struct with hardcoded data.

## Implementation (2026-03-07)

**Design decision**: `.onAppear` re-query (Approach A) — lowest blast radius, single-file change.

### Changes to `ContentView.swift`

1. **`LibraryView` now accepts `podcastManager: PodcastManaging`** — dependency threaded from `ContentView` initializer at line 535.
2. **`loadData()` calls `podcastManager.all()`** — replaced hardcoded `samplePodcasts` array.
3. **`.onAppear` triggers `loadData()`** — refreshes every time the Library tab becomes visible (handles Discover→Library navigation).
4. **`ContentUnavailableView` empty state** — shown when `podcastManager.all()` returns empty, with accessibility identifier `"Library.EmptyState"`.
5. **`UITEST_SEED_PODCASTS` environment guard** — when set to `"1"` and no podcasts exist, seeds a "Swift Talk" sample podcast so UI tests can navigate Library without requiring a live RSS feed.
6. **Removed dead `@Query private var items: [Item]`** — Xcode SwiftData template scaffolding that became dead code after the wiring was complete.

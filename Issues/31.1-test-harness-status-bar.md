# Issue 31.1: Persistent Status Bar for run-xcode-tests.sh

**Status**: Open
**Priority**: Medium
**Type**: Developer Experience / Tooling
**Effort**: Small (2–4 hours)
**GitHub Issue**: [#425](https://github.com/ezigus/zpod/issues/425)

## Description

Add a persistent status line at the bottom of the terminal that remains visible while `./scripts/run-xcode-tests.sh` scrolls output. The bar shows the current pipeline stage, total elapsed time, and a live tally of test results (total run, passed, failed, skipped).

## Problem Statement

The test harness runs multi-phase pipelines (syntax → build → UI suites → lint) that can take 20–30 minutes. Currently there is no at-a-glance way to know:
- Which phase is currently executing
- How long the overall run has been going
- How many tests have passed or failed so far

The information is buried in scrolling log output and only summarized at the very end.

## Acceptance Criteria

- [ ] A single fixed line appears at the bottom of the terminal throughout the run
- [ ] Line updates in real time (~0.5 s interval) showing:
  - Current phase/stage name (e.g., "UI tests SwipeActionManagementTests")
  - Elapsed wall-clock time (e.g., ⏱ 2m14s)
  - Cumulative counts: ✅ passed  ❌ failed  ⏭ skipped  📋 total
- [ ] Normal log output scrolls above the bar undisturbed
- [ ] Bar disappears cleanly before the final summary is printed
- [ ] **No-op when piped or in CI** (`$CI` set or `[[ ! -t 1 ]]`)
- [ ] No effect on exit codes, test results, or log file content
- [ ] Terminal scroll region is fully restored on exit (including Ctrl-C / ERR)

## Technical Design

### New file: `scripts/lib/statusline.sh`

Self-contained library sourced by `run-xcode-tests.sh`. Uses VT100 scroll-region escape codes to pin the last terminal line:

```
\033[1;{ROWS-1}r    ← restrict scrolling to rows 1..ROWS-1
\033[{ROWS};1H      ← position cursor on last row
\033[7m…\033[m      ← render bar in reverse-video
\0337 / \0338       ← save/restore cursor around each write
```

A background subshell (`_statusline_updater_loop`) reads a shared state file every 0.5 s and rewrites the bar. State is updated via atomic `sed > tmp && mv tmp state` writes.

### Hooks in `scripts/lib/harness.sh` (4 additions)

| Location | Addition |
|----------|----------|
| `begin_phase()` end | `statusline_set_phase "$1"` |
| `add_summary()` end | `statusline_record_result "$total" "$passed" "$failed" "$skipped"` |
| `finalize_and_exit()` first line | `statusline_cleanup` |
| `harness_main()` after `START_TIME=$(date +%s)` | `statusline_init` |

### Visual format

```
  UI tests SwipeActionManagementTests    ⏱ 2m14s  ✅ 4  ❌ 0  ⏭ 0  📋 4 run
```
Rendered in reverse-video on the last terminal row; truncated gracefully when the terminal is narrow.

## Dependencies

- None — pure bash with `tput` (available on all macOS systems)

## Testing Strategy

1. Run targeted test: confirm bar appears, updates, and cleans up before final summary
2. Run syntax-only (`-s`): confirm bar works for non-test phases
3. Pipe output (`| cat`): confirm zero VT100 sequences emitted
4. Interrupt with Ctrl-C: confirm terminal scroll region is restored
5. Verify existing test suite still passes (no exit code regression)
